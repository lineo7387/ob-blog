oh-blog 基础架构方案

请比较模块边界、未来可扩展性和第一版复杂度。可以点击一个方案，也可以直接在终端回复。

A. 极简本地博客
app/ routes
content/posts/*.mdx
lib/posts.ts 读取 + 排序
components/* 展示

A. 极简本地博客

依赖最少、最快落地。适合第一版文章不多、功能简单，但未来替换 CMS 时页面会更容易被数据细节牵住。

B. 推荐：内容抽象层
app/ routes
features/blog UI
components/design-system
lib/content API
content/posts/*.mdx

B. 内容抽象层

页面只依赖稳定的 Post 模型和查询函数。第一版仍然轻量，后续换 CMS、加标签、加搜索更顺滑。

C. 内容平台雏形
app/ + api + admin-ready
content repository
tag/search indexes
route handlers
preview/draft hooks

C. 内容平台雏形

扩展性最强，但第一版会引入更多没马上用到的边界。适合已确定要做后台、搜索、草稿预览。

我的推荐

  • 选 B：和你前面选的“本地抽象层，未来可换 CMS”一致。
  • 保留根目录 app/，共享代码按责任放 components/、features/、lib/、content/。
  • 用 Vitest 覆盖 content API 和纯组件，避免基础设施悄悄坏掉。

需要克制

  • 不提前做后台、搜索、标签页、评论、订阅系统。
  • 不把整个站点标成 Client Component。
  • 不把 Vaporwave 效果散落成一次性 class，先沉到 token 和基础组件。
