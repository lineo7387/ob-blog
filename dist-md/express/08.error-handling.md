# Express 错误处理

任何生产级应用都必须具备健壮的错误处理机制。Express 提供了一套专门的错误处理中间件系统，用于捕获应用运行过程中出现的各类异常，并以统一、友好的方式返回给客户端。

## 错误处理中间件基础

### 识别错误处理中间件

错误处理中间件与普通中间件的签名区别在于它有**四个参数**，第一个参数为 `err`（错误对象）：

```js
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ error: '服务器内部错误' });
});
```

Express 通过函数形参数量来判断中间件类型。即使不使用 `next`，也必须将其声明为第四个参数。

### 错误传递路径

当应用中的任何位置调用 `next(err)` 或抛出未捕获异常时，Express 会跳过所有正常的中间件和路由处理器，直接进入第一个匹配的错误处理中间件：

```js
app.get('/data', (req, res, next) => {
  const data = JSON.parse(req.query.json);
  // 如果 JSON.parse 抛出 SyntaxError，会被错误处理中间件捕获
  res.json(data);
});

// 错误处理中间件（必须放在所有路由之后）
app.use((err, req, res, next) => {
  if (err instanceof SyntaxError) {
    return res.status(400).json({ error: '无效的 JSON 格式' });
  }
  next(err); // 传递给其他错误处理中间件
});
```

## 同步错误的捕获

### 路由处理器中的 try/catch

```js
app.get('/divide', (req, res, next) => {
  try {
    const a = parseInt(req.query.a);
    const b = parseInt(req.query.b);

    if (b === 0) {
      throw new Error('除数不能为零');
    }

    res.json({ result: a / b });
  } catch (err) {
    next(err); // 将错误传递给错误处理中间件
  }
});
```

### 自定义错误类

通过继承 `Error` 类，可以创建语义化的业务错误，便于错误处理中间件分类响应：

```js
// errors/AppError.js
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
    this.status = `${statusCode}`.startsWith('4') ? 'fail' : 'error';
    this.isOperational = true; // 标记为可预期的业务错误

    Error.captureStackTrace(this, this.constructor);
  }
}

module.exports = AppError;
```

```js
// 使用自定义错误
const AppError = require('./errors/AppError');

app.get('/users/:id', (req, res, next) => {
  const user = findUserById(req.params.id);

  if (!user) {
    return next(new AppError('用户不存在', 404));
  }

  res.json(user);
});
```

## 异步错误的统一处理

### 异步函数中的错误陷阱

Express 4.x 不会自动捕获 `async` 函数中抛出的异常：

```js
// ❌ 错误：异常不会被 Express 捕获，导致进程崩溃
app.get('/users', async (req, res) => {
  const users = await db.query('SELECT * FROM users'); // 如果查询失败，应用崩溃
  res.json(users);
});
```

### 方案一：显式 try/catch

```js
app.get('/users', async (req, res, next) => {
  try {
    const users = await db.query('SELECT * FROM users');
    res.json(users);
  } catch (err) {
    next(err);
  }
});
```

每个异步路由都写 try/catch 会导致大量重复代码。

### 方案二：包装器函数（推荐）

创建一个高阶函数，自动捕获异步函数中的错误并调用 `next(err)`：

```js
// utils/catchAsync.js
const catchAsync = (fn) => {
  return (req, res, next) => {
    fn(req, res, next).catch(next);
  };
};

module.exports = catchAsync;
```

```js
const catchAsync = require('./utils/catchAsync');

// ✅ 简洁且安全
app.get('/users', catchAsync(async (req, res) => {
  const users = await db.query('SELECT * FROM users');
  res.json(users);
}));

app.post('/users', catchAsync(async (req, res) => {
  const newUser = await db.createUser(req.body);
  res.status(201).json(newUser);
}));
```

### 方案三：Express 5.x 原生支持

Express 5.x 起，异步函数中 rejected 的 Promise 会被自动转发到错误处理中间件，无需额外包装：

```js
// Express 5.x —— 自动捕获 async 错误
app.get('/users', async (req, res) => {
  const users = await db.query('SELECT * FROM users');
  res.json(users);
});
```

这是 Express 5 的重大改进之一，也是推荐升级的理由。

## 全局错误处理中间件

### 生产级错误响应

```js
// middleware/errorHandler.js
const AppError = require('../errors/AppError');

const errorHandler = (err, req, res, next) => {
  err.statusCode = err.statusCode || 500;
  err.status = err.status || 'error';

  // 开发环境：返回完整错误信息
  if (process.env.NODE_ENV === 'development') {
    return res.status(err.statusCode).json({
      status: err.status,
      error: err,
      message: err.message,
      stack: err.stack
    });
  }

  // 生产环境：过滤敏感信息
  if (err.isOperational) {
    // 可预期的业务错误
    return res.status(err.statusCode).json({
      status: err.status,
      message: err.message
    });
  }

  // 未知的编程错误，不暴露细节
  console.error('未预期错误:', err);
  return res.status(500).json({
    status: 'error',
    message: '服务器内部错误'
  });
};

module.exports = errorHandler;
```

```js
// app.js
const errorHandler = require('./middleware/errorHandler');

// 所有路由之后挂载
app.use(errorHandler);
```

### 404 处理

当没有任何路由匹配请求时，应返回 404 错误。这通常通过一个放在路由定义之后、错误处理中间件之前的 catch-all 路由实现：

```js
// 在所有路由之后
app.all('*', (req, res, next) => {
  next(new AppError(`找不到路径 ${req.originalUrl}`, 404));
});

// 错误处理中间件
app.use(errorHandler);
```

> `app.all()` 匹配所有 HTTP 方法，`'*'` 匹配所有路径。

## 常见错误场景处理

### 验证错误（Joi / Zod）

```js
const { z } = require('zod');

const userSchema = z.object({
  name: z.string().min(1),
  email: z.string().email()
});

app.post('/users', (req, res, next) => {
  const result = userSchema.safeParse(req.body);

  if (!result.success) {
    const message = result.error.errors.map(e => e.message).join(', ');
    return next(new AppError(message, 400));
  }

  res.status(201).json(result.data);
});
```

### 数据库连接错误

```js
// 启动时检查数据库连接
db.connect()
  .then(() => {
    app.listen(3000, () => console.log('服务启动'));
  })
  .catch(err => {
    console.error('数据库连接失败:', err.message);
    process.exit(1); // 无法正常启动，退出进程
  });
```

### 进程级未捕获异常

Express 错误处理中间件只能捕获请求生命周期内的错误。对于进程级别的异常，需要额外处理：

```js
// 捕获未处理的 Promise 拒绝
process.on('unhandledRejection', (err) => {
  console.error('未处理的 Promise 拒绝:', err.name, err.message);
  // 优雅关闭服务器，给正在处理的请求时间完成
  server.close(() => process.exit(1));
});

// 捕获未捕获的异常
process.on('uncaughtException', (err) => {
  console.error('未捕获的异常:', err.name, err.message);
  server.close(() => process.exit(1));
});
```

## 错误处理最佳实践

1. **始终将错误处理中间件放在路由之后**：否则它不会生效
2. **区分业务错误与程序错误**：业务错误返回友好的客户端消息，程序错误记录日志并返回通用提示
3. **不要在生产环境暴露堆栈**：防止泄露代码结构
4. **使用统一的错误响应格式**：便于前端统一处理

```json
{
  "status": "fail",
  "message": "用户不存在"
}
```

5. **记录所有错误日志**：结合 `morgan` 和 `winston` 等日志库追踪问题

## 小结

本章全面覆盖了 Express 错误处理的各个层面：

- **错误处理中间件**：四参数签名，Express 自动识别
- **同步错误**：`try/catch` + `next(err)` 显式传递
- **自定义错误类**：`AppError` 区分业务错误与系统错误
- **异步错误**：`catchAsync` 包装器或 Express 5.x 自动捕获
- **全局处理**：开发环境返回完整信息，生产环境过滤敏感内容
- **404 处理**：`app.all('*')` 捕获未匹配路由
- **进程安全**：`unhandledRejection` 与 `uncaughtException` 兜底

建立统一的错误处理体系后，应用在面对异常时能够保持稳定运行，并为客户端提供清晰、一致的反馈。下一章将学习 Express 与模板引擎的集成，实现服务端渲染页面。
