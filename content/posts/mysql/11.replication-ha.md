# MySQL 主从复制与高可用

单节点 MySQL 存在单点故障风险，也无法分担读请求压力。**主从复制**（Replication）是构建高可用数据库架构的基础。本文将讲解复制原理、配置方法、GTID 机制，以及读写分离方案。

---

## 主从复制原理

MySQL 主从复制基于 **binlog**（二进制日志）实现，核心流程分为三个步骤：

1. **主库记录**：主库将数据变更写入 binlog
2. **从库拉取**：从库的 I/O 线程连接主库，拉取 binlog 并写入中继日志（relay log）
3. **从库重放**：从库的 SQL 线程读取 relay log，在从库上重放这些变更

```
Master                      Slave
  |                           |
  |  1. 写 binlog             |
  |  <----------------------  |  2. I/O 线程请求
  |  发送 binlog              |
  |  ---------------------->  |
  |                           |  3. 写入 relay log
  |                           |  4. SQL 线程重放
```

### binlog 格式

MySQL binlog 有三种格式：

| 格式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| `STATEMENT` | 记录 SQL 语句 | 日志量小 | 某些语句可能导致主从不一致 |
| `ROW` | 记录行变更 | 精确、安全 | 日志量大 |
| `MIXED` | 混合模式 | 自动选择 | 逻辑复杂 |

MySQL 5.7+ 默认使用 `ROW` 格式，推荐生产环境使用：

```sql
-- 查看当前 binlog 格式
SHOW VARIABLES LIKE 'binlog_format';

-- 设置为 ROW 格式
SET GLOBAL binlog_format = 'ROW';
```

## 配置异步复制

**异步复制**是 MySQL 默认的复制模式，主库写入 binlog 后立即返回，不等待从库确认。

### 主库配置

编辑 `my.cnf`：

```ini
[mysqld]
server-id = 1
log_bin = /var/lib/mysql/mysql-bin
binlog_format = ROW
# MySQL 8.0+ 使用 binlog_expire_logs_seconds 代替 expire_logs_days（7天 = 604800秒）
binlog_expire_logs_seconds = 604800
max_binlog_size = 500M

# 需要复制的数据库（可选）
binlog_do_db = mydb
# 或排除的数据库
# binlog_ignore_db = test
```

重启 MySQL 后创建复制用户：

```sql
CREATE USER 'repl'@'%' IDENTIFIED BY 'repl_password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';
FLUSH PRIVILEGES;

-- 查看主库状态
SHOW MASTER STATUS;
-- 记录 File 和 Position，如：mysql-bin.000001, 154
```

### 从库配置

编辑 `my.cnf`：

```ini
[mysqld]
server-id = 2
relay_log = /var/lib/mysql/mysql-relay-bin
log_bin = /var/lib/mysql/mysql-bin
read_only = 1
```

配置复制连接：

> **MySQL 8.0.23+ 术语变更**：`CHANGE MASTER TO` 更名为 `CHANGE REPLICATION SOURCE TO`，`START SLAVE` 更名为 `START REPLICA`，`SHOW SLAVE STATUS` 更名为 `SHOW REPLICA STATUS`。旧语法仍兼容但已弃用。

```sql
CHANGE MASTER TO
  MASTER_HOST = 'master_host',
  MASTER_PORT = 3306,
  MASTER_USER = 'repl',
  MASTER_PASSWORD = 'repl_password',
  MASTER_LOG_FILE = 'mysql-bin.000001',
  MASTER_LOG_POS = 154;

START SLAVE;

-- 查看从库状态
SHOW SLAVE STATUS\G
```

关键状态字段：

- `Slave_IO_Running`：I/O 线程是否运行
- `Slave_SQL_Running`：SQL 线程是否运行
- `Seconds_Behind_Master`：从库延迟秒数
- `Last_IO_Error` / `Last_SQL_Error`：错误信息

## 半同步复制

**异步复制**的问题是：如果主库崩溃，部分已提交的数据可能尚未同步到从库，导致数据丢失。

**半同步复制**（Semi-Synchronous Replication）要求主库至少等待一个从库确认收到 binlog 后才返回成功，兼顾性能和数据安全。

### 配置半同步复制

```sql
-- 主库安装插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';

-- 从库安装插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 主库启用
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_master_timeout = 10000; -- 10秒超时降级为异步

-- 从库启用
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
STOP SLAVE IO_THREAD;
START SLAVE IO_THREAD;

-- 验证状态
SHOW STATUS LIKE 'Rpl_semi_sync_master_status';
SHOW STATUS LIKE 'Rpl_semi_sync_slave_status';
```

## GTID 复制

**GTID**（Global Transaction Identifier）是 MySQL 5.6+ 引入的全局事务标识符，格式为 `UUID:事务序号`。

### GTID 优势

- **自动定位**：无需手动指定 `MASTER_LOG_FILE` 和 `MASTER_LOG_POS`
- **故障切换简单**：新主库可以自动跳过已执行的事务
- **一致性保证**：避免主从数据不一致

### 配置 GTID 复制

主从库 `my.cnf` 都添加：

```ini
[mysqld]
gtid_mode = ON
enforce_gtid_consistency = ON
log_bin = /var/lib/mysql/mysql-bin
server-id = 1  -- 从库改为 2
```

从库配置：

```sql
CHANGE MASTER TO
  MASTER_HOST = 'master_host',
  MASTER_PORT = 3306,
  MASTER_USER = 'repl',
  MASTER_PASSWORD = 'repl_password',
  MASTER_AUTO_POSITION = 1;

START SLAVE;

-- 查看 GTID 执行情况
SHOW SLAVE STATUS\G
-- Retrieved_Gtid_Set：已从主库接收的 GTID
-- Executed_Gtid_Set：已在从库执行的 GTID
```

## 读写分离

主从架构最常见的用途是**读写分离**：写操作走主库，读操作走从库。

### 应用层实现

在应用代码中配置多个数据源，根据 SQL 类型路由：

```java
// Spring Boot 多数据源示例配置
@Configuration
public class DataSourceConfig {
    @Bean
    @Primary
    public DataSource masterDataSource() {
        return DataSourceBuilder.create()
            .url("jdbc:mysql://master:3306/mydb")
            .build();
    }

    @Bean
    public DataSource slaveDataSource() {
        return DataSourceBuilder.create()
            .url("jdbc:mysql://slave:3306/mydb")
            .build();
    }
}
```

### ProxySQL 实现读写分离

**ProxySQL** 是高性能的 MySQL 代理中间件，支持读写分离、连接池、查询缓存和故障切换。

#### 安装 ProxySQL

```bash
# Ubuntu/Debian
apt install proxysql
systemctl start proxysql

# 默认管理端口 6032，服务端口 6033
mysql -u admin -padmin -h 127.0.0.1 -P 6032
```

#### 配置后端服务器

```sql
-- 添加主库和从库
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight)
VALUES (10, 'master_host', 3306, 1);   -- 写组

INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight)
VALUES (20, 'slave1_host', 3306, 1),   -- 读组
       (20, 'slave2_host', 3306, 1);

LOAD MYSQL SERVERS TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;
```

#### 配置读写分离规则

```sql
-- 配置用户
INSERT INTO mysql_users (username, password, default_hostgroup)
VALUES ('app_user', 'app_password', 10);

-- 配置查询路由规则：SELECT 走读组
INSERT INTO mysql_query_rules (rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (1, 1, '^SELECT.*FOR UPDATE$', 10, 1),  -- SELECT FOR UPDATE 走主库
       (2, 1, '^SELECT', 20, 1);                 -- 普通 SELECT 走从库

LOAD MYSQL USERS TO RUNTIME;
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL USERS TO DISK;
SAVE MYSQL QUERY RULES TO DISK;
```

#### 监控复制延迟

```sql
-- 在 ProxySQL 中查看服务器状态
SELECT * FROM monitor.mysql_server_replication_lag_log ORDER BY time_start_us DESC LIMIT 10;

-- 配置延迟阈值（超过 1 秒不从该从库读）
UPDATE mysql_servers SET max_replication_lag = 1 WHERE hostgroup_id = 20;
LOAD MYSQL SERVERS TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;
```

### HAProxy 负载均衡

**HAProxy** 是另一种常用的负载均衡方案，配合 Keepalived 实现高可用。

```bash
# /etc/haproxy/haproxy.cfg 示例
global
    daemon
    maxconn 4096

defaults
    mode tcp
    timeout connect 5s
    timeout client 30s
    timeout server 30s

frontend mysql_read
    bind *:3307
    default_backend mysql_slaves

backend mysql_slaves
    balance roundrobin
    option mysql-check user haproxy_check
    server slave1 slave1_host:3306 check
    server slave2 slave2_host:3306 check
```

## 高可用方案

### MHA（Master High Availability）

MHA 是成熟的 MySQL 高可用方案，能在主库故障时快速（通常 30 秒内）自动切换到从库。

主要组件：
- **MHA Manager**：监控主库状态，执行故障切换
- **MHA Node**：运行在每个 MySQL 服务器上，执行日志解析和 relay log 差异应用

### MySQL InnoDB Cluster

MySQL 官方提供的高可用方案，基于 **Group Replication**：

- 自动故障检测和切换
- 数据强一致性（多数派确认）
- 内置读写分离（MySQL Router）

```bash
# 使用 MySQL Shell 配置 InnoDB Cluster
mysqlsh

# 配置实例
dba.configureInstance('root@node1:3306');
dba.configureInstance('root@node2:3306');
dba.configureInstance('root@node3:3306');

# 创建集群
var cluster = dba.createCluster('myCluster');
cluster.addInstance('root@node2:3306');
cluster.addInstance('root@node3:3306');

# 查看集群状态
cluster.status();
```

## 复制延迟处理

从库延迟是主从架构的常见问题。

### 查看延迟

```sql
-- 方法1：Seconds_Behind_Master
SHOW SLAVE STATUS\G

-- 方法2：对比主从的 GTID 或 binlog 位置
-- 主库
SHOW MASTER STATUS;
-- 从库
SHOW SLAVE STATUS;
```

### 减少延迟的方法

1. **硬件升级**：从库使用 SSD、增加内存
2. **并行复制**：MySQL 5.6+ 支持基于库的并行复制，5.7+ 支持基于事务组的并行复制

```ini
# my.cnf 配置并行复制
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 4
```

3. **拆分大事务**：大事务会导致从库长时间阻塞，应尽量拆分为小事务
4. **读写分离策略**：延迟较大时，将读请求临时切回主库

---

## 小结

- **异步复制**配置简单，但存在数据丢失风险
- **半同步复制**在主库等待从库确认，平衡了性能和安全
- **GTID** 简化了复制管理和故障切换
- **ProxySQL** 是实现读写分离和连接池的推荐中间件
- **InnoDB Cluster** 是 MySQL 官方推荐的完整高可用方案

主从复制是 MySQL 扩展读能力和保障可用性的基石，配合合适的监控和切换策略，可以构建稳定可靠的数据库架构。
