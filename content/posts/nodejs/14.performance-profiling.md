# 性能剖析与内存优化

性能是后端服务的生命线。Node.js 应用在高负载下可能出现 CPU 飙高、内存泄漏、响应延迟等问题。掌握性能剖析和内存优化的方法，能够帮助你快速定位瓶颈，写出高效的代码。

## V8 堆内存结构

Node.js 的内存由 V8 引擎管理，主要分为两个区域：

### 新生代（Young Generation）

存放生命周期短的对象，分为 **From** 和 **To** 两个空间。新创建的对象先放在 From 空间，垃圾回收时将存活对象复制到 To 空间，然后交换两个空间。

### 老生代（Old Generation）

存放经过多次垃圾回收仍然存活的对象。老生代使用 **标记-清除（Mark-Sweep）** 和 **标记-整理（Mark-Compact）** 算法。

```js
// 查看内存使用情况
console.log(process.memoryUsage());
// {
//   rss: 35848192,        // 进程常驻内存
//   heapTotal: 6537216,   // V8 堆总大小
//   heapUsed: 4432104,    // 已使用堆内存
//   external: 1234567,    // C++ 对象绑定内存
//   arrayBuffers: 234567  // ArrayBuffer 内存
// }
```

## 垃圾回收机制

V8 使用**分代垃圾回收**策略，针对不同生命周期的对象采用不同的回收算法。

### Scavenge（新生代回收）

采用 **Cheney 算法**，复制存活对象，时间复杂度与存活对象数量成正比。速度快、频率高。

### Mark-Sweep & Mark-Compact（老生代回收）

- **标记阶段**：从根对象（全局对象、执行栈）出发，遍历所有可达对象并标记
- **清除阶段**：回收未标记的对象内存
- **整理阶段**：将存活对象向一端移动，减少内存碎片

```js
// 强制触发垃圾回收（需要 --expose-gc 标志）
node --expose-gc app.js

// 代码中手动触发（仅用于调试）
if (global.gc) {
  global.gc();
  console.log('GC 后:', process.memoryUsage().heapUsed);
}
```

## 内存泄漏常见原因

内存泄漏是 Node.js 应用最常见的问题之一。以下是几种典型场景。

### 1. 全局变量累积

```js
// 错误：全局数组无限增长
const leaks = [];

function handleRequest(req, res) {
  leaks.push(req.body); // 永远不会释放
  res.end('OK');
}
```

### 2. 闭包引用

```js
// 错误：闭包持有大对象引用
function createHandler() {
  const bigData = new Array(1000000).fill('x');

  return function handler(req, res) {
    // 即使不使用 bigData，闭包仍然持有引用
    res.end('OK');
  };
}

const handler = createHandler();
// handler 存活期间，bigData 无法被回收
```

### 3. 事件监听器未移除

```js
const EventEmitter = require('events');
const emitter = new EventEmitter();

// 错误：每次请求都添加监听器，从不移除
function handleRequest(req, res) {
  emitter.on('data', (data) => {
    res.write(data);
  });
  res.end('OK');
}

// 正确：使用 once 或及时移除
function handleRequestFixed(req, res) {
  const onData = (data) => res.write(data);
  emitter.once('data', onData);

  req.on('close', () => {
    emitter.removeListener('data', onData);
  });
}
```

### 4. 定时器未清理

```js
// 错误：setInterval 永不停止
function startMonitoring() {
  setInterval(() => {
    console.log(process.memoryUsage());
  }, 1000);
}

// 正确：保存引用以便停止
let monitorInterval;
function startMonitoringFixed() {
  monitorInterval = setInterval(() => {
    console.log(process.memoryUsage());
  }, 1000);
}

function stopMonitoring() {
  clearInterval(monitorInterval);
}
```

### 5. 缓存无上限

```js
// 错误：Map 无限增长
const cache = new Map();

function getData(key) {
  if (cache.has(key)) {
    return cache.get(key);
  }
  const data = fetchFromDatabase(key);
  cache.set(key, data); // 永不清除
  return data;
}

// 正确：使用 LRU 缓存
const LRU = require('lru-cache');
const cache = new LRU({ max: 1000, ttl: 1000 * 60 * 5 });
```

## --prof 性能分析

Node.js 内置了 V8 性能分析器，通过 `--prof` 标志生成日志文件。

```bash
# 运行应用并收集性能数据
node --prof app.js

# 压测（另开终端）
npx autocannon -c 100 -d 30 http://localhost:3000

# 停止应用后，生成 v8.log 文件
# 处理日志生成可读报告
node --prof-process isolate-0x*.log > profile.txt
```

`profile.txt` 包含：

- **共享库**：系统库和 Node.js 二进制文件的耗时
- **JavaScript**：你的代码耗时
- **C++**：V8 运行时耗时
- **GC**：垃圾回收耗时

## Chrome DevTools 火焰图

对于更直观的分析，可以使用 Chrome DevTools 的 Performance 面板。有两种方式：实时录制或加载预生成的 profile 文件。

### 方式一：实时录制（需 `--inspect`）

```bash
# 启动应用并开启调试协议
node --inspect app.js
```

在 Chrome DevTools 中：

1. 打开 `chrome://inspect`
2. 点击 "Open dedicated DevTools for Node"
3. 切换到 **Performance** 面板
4. 点击录制按钮，执行操作后停止
5. 查看火焰图，找出耗时最长的函数

### 方式二：生成 CPU Profile 文件（`--cpu-prof`）

```bash
# 运行应用并自动生成 CPU profile 文件
node --cpu-prof --cpu-prof-dir=./profiles app.js
```

停止应用后，`./profiles` 目录会生成 `.cpuprofile` 文件。在 Chrome DevTools 的 **Performance** 面板中右键选择 "Load profile..." 即可查看火焰图。

### 程序化生成 CPU Profile

```js
const fs = require('fs');
const inspector = require('inspector');
const session = new inspector.Session();
session.connect();

function startCpuProfile(name) {
  session.post('Profiler.enable', () => {
    session.post('Profiler.start', () => {
      console.log(`开始 CPU Profile: ${name}`);
    });
  });
}

function stopCpuProfile(filename) {
  session.post('Profiler.stop', (err, { profile }) => {
    if (err) throw err;
    fs.writeFileSync(filename, JSON.stringify(profile));
    console.log(`CPU Profile 已保存: ${filename}`);
    session.disconnect();
  });
}

// 使用
startCpuProfile('heavy-operation');

setTimeout(() => {
  stopCpuProfile('./cpu-profile.cpuprofile');
}, 10000);
```

## 内存分析

### 堆快照

```js
const v8 = require('v8');
const fs = require('fs');

// 生成堆快照
const snapshotStream = v8.getHeapSnapshot();
const fileStream = fs.createWriteStream('heap.heapsnapshot');
snapshotStream.pipe(fileStream);

fileStream.on('finish', () => {
  console.log('堆快照已保存');
});
```

在 Chrome DevTools 的 **Memory** 面板中加载 `.heapsnapshot` 文件，可以：

- 查看对象数量和大小
- 追踪对象引用链
- 对比多个快照找出泄漏


## eventLoopUtilization 监控

Node.js 提供了 `perf_hooks` 模块来监控事件循环的健康状况。

```js
const { eventLoopUtilization } = require('perf_hooks').performance;
const { monitorEventLoopDelay } = require('perf_hooks');

// 监控事件循环利用率
let lastELU = eventLoopUtilization();

setInterval(() => {
  const elu = eventLoopUtilization(lastELU);
  console.log(`事件循环利用率: ${(elu.utilization * 100).toFixed(2)}%`);
  console.log(`事件循环延迟: ${elu.idle}ms idle, ${elu.active}ms active`);
  lastELU = elu;
}, 5000);

// 监控事件循环延迟
const h = monitorEventLoopDelay({ resolution: 20 });
h.enable();

setInterval(() => {
  console.log(`\n事件循环延迟统计:`);
  console.log(`  最小: ${h.min}ns`);
  console.log(`  最大: ${h.max}ns`);
  console.log(`  平均: ${h.mean}ns`);
  console.log(`  第 99 百分位: ${h.percentile(99)}ns`);
  h.reset();
}, 10000);
```

高事件循环延迟通常意味着：

- 同步 CPU 密集型任务阻塞了事件循环
- 大量微任务累积
- 垃圾回收耗时过长

## 性能优化实践

### 1. 使用 Stream 处理大数据

```js
// 错误：一次性读取大文件
const data = fs.readFileSync('huge-file.txt'); // 内存爆炸

// 正确：流式处理
fs.createReadStream('huge-file.txt')
  .pipe(transformStream)
  .pipe(fs.createWriteStream('output.txt'));
```

### 2. 避免在热路径使用 JSON.stringify

```js
// 如果频繁序列化相同对象，可以缓存结果
const cache = new Map();

function cachedStringify(obj) {
  const key = obj.id;
  if (cache.has(key)) {
    return cache.get(key);
  }
  const str = JSON.stringify(obj);
  cache.set(key, str);
  return str;
}
```

### 3. 使用 Buffer 池减少内存分配

```js
// 高频创建 Buffer 时考虑复用
const BufferPool = require('buffer-pool'); // 或使用自定义实现
```

### 4. 设置合理的内存限制

```bash
# 限制堆内存大小（默认根据系统内存动态调整）
node --max-old-space-size=4096 app.js

# 限制新生代大小
node --max-semi-space-size=64 app.js
```

## 总结

Node.js 性能优化是一门系统工程：

- 理解 V8 的分代垃圾回收机制，新生代 Scavenge、老生代 Mark-Sweep/Compact
- 内存泄漏的常见原因：全局变量、闭包、事件监听器、定时器、无界缓存
- `--prof` 和 `--prof-process` 进行命令行性能分析
- Chrome DevTools 提供可视化的火焰图和堆分析
- `clinic.js` 提供一站式的性能诊断方案
- `eventLoopUtilization` 和 `monitorEventLoopDelay` 监控事件循环健康
- Stream、缓存、Buffer 池是日常优化的有效手段

定期进行性能剖析，建立监控基线，才能在问题发生前发现隐患。
